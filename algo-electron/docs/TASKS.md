# AI Coach 任务路线图

> 本文档是 AI Coach 从 MVP 到复赛创新的精细任务拆解，后续按此执行。配套技术选型见 [ai coach技术栈.md](./ai%20coach技术栈.md)。

## 1. 目标

把 AI Coach 做成：

> **科技感视觉桌宠 + 本地学习教练 + 可选 LLM/RAG 增强** 的组合，而不是普通聊天框。

核心体验：

- 桌面上常驻一个类 Codex 科技感小人，根据做题状态给出低打扰提示。
- 默认本地离线运行，不调用大模型。
- 触发信号以平台实时抓取的提交结果为主。
- 用户卡住时由用户主动上传/粘贴代码后才进入代码分析。
- 视觉效果与可自定义形象是产品差异化重点。
- 大模型、RAG、向量库均为可选增强，不做 MVP 必需项。
- rated 比赛进行中自动进入比赛模式，绝对静默，可事后证明（合规红线，见 T2.6）。

## 2. 产品原则

| 原则 | 说明 |
|---|---|
| 视觉优先 | 创新视觉效果是首要目标，先做出“不像聊天框”的科技感桌宠。 |
| 本地离线优先 | 默认不调用云 LLM；规则、模板、提示库都在本地。 |
| 提交结果驱动 | 以实时抓取到的 verdict 作为主要事件源，不假设能持续读取用户代码。 |
| 用户主动上传代码 | 代码分析是用户主动触发的入口，不做后台静默采集源码。 |
| 标签仅供参考 | 部分平台不提供算法标签，所有 MVP 判断不得依赖标签。 |
| 无大模型也能用 | 默认只给通用小提示（复杂度、边界、数据范围等），不强制接 LLM。 |
| 不直接给答案 | 提示分级，不输出完整题解。 |
| 不打扰 | 默认低频、低强度，用户可关闭。 |
| 比赛红线 | rated 比赛进行中 Coach 绝对静默：自动检测、默认开启、不可绕过，并留可导出的审计日志（CF 2024-09 规则明令赛中禁 AI）。 |

## 3. 里程碑拆分

| 里程碑 | 目标 | 关键产出 |
|---|---|---|
| M0 | 文档与架构确认 | 本文档 + 技术栈文档 |
| M1 | 桌宠视觉壳与状态系统 | 透明窗口 + 科技感小人 + 状态切换 |
| M2 | Coach 事件与本地规则触发 | 从提交结果/停留时间生成 CoachEvent + 比赛模式硬保障 |
| M3 | 通用提示库与低 token 交互 | 复杂度/边界/数据范围等小提示 + 本地题面约束解析 |
| M4 | 用户主动上传代码分析入口 | 手动入口 + 隐私提示 + 可选 LLM |
| M5 | 个性化形象生成/导入 | 用户自定义科技感形象 |
| M6 | 可选 LLM Provider | OpenAI-compatible 抽象 + DeepSeek/Qwen/Ollama |
| M7 | 知识库/RAG | 模板库、题型卡、本地检索 |
| M8 | 复赛创新增强 | 错因归因、学习者图谱、情绪系统、Socratic Ladder |
| M9 | 过程复盘与教练报表 | 单题时间轴、虚拟赛复盘、数据导出、教练周报、效果指标页 |

### M1：桌宠视觉壳与状态系统

#### T1.1 创建 Coach 桌宠透明窗口

- 目标：在主窗口之外新增一个透明、置顶、可点击穿透的 BrowserWindow 作为桌宠容器。
- 涉及文件（新增）：
  - `electron/coach/CoachPetWindow.ts`
  - `electron/coach/README.md`
- 实现要点：
  - `BrowserWindow` 配置：`transparent: true`、`frame: false`、`alwaysOnTop: true`、`skipTaskbar: true`、`hasShadow: false`、`resizable: false`。
  - 窗口内容加载与主窗口同源：开发环境 `loadURL(devServerUrl + '#/coach-pet')`，生产 `loadFile(index.html, { hash: 'coach-pet' })`，renderer 按 hash 分流渲染桌宠入口。
  - preload 复用主窗口的 `preload.mjs`（与 `main.ts` 中 `path.join(__dirname, 'preload.mjs')` 一致），只暴露白名单 API。
  - 支持点击穿透切换：默认穿透，气泡/按钮等可交互区域 hover 时经 IPC 临时关闭穿透，离开后恢复（`setIgnoreMouseEvents(true, { forward: true })`，`forward` 仅 Win/macOS）。
  - 拖拽移动支持。
  - 与主窗口生命周期绑定：主窗口关闭时桌宠也退出。
  - 入口在 `electron/main.ts` 中初始化。
- 完成标准：桌面上出现一个透明悬浮窗口，不影响主应用使用。
- 验证方式：本地启动应用观察。
- 优先级：P0。

#### T1.2 实现科技感小人渲染层

- 目标：在桌宠窗口中渲染一个类 Codex 风格的科技感小人。
- 涉及文件（新增）：

- 实现要点：

  - 状态枚举：`idle / thinking / alert / celebrate / sleep / focus`。
  - 每个状态有对应动画与配色。
  - 不要做成萌系宠物，保持冷感、极简、数据流光感。
- 完成标准：可切换状态并观察到视觉变化。
- 验证方式：手动切换状态观察。
- 优先级：P0。

#### T1.3 实现气泡与交互按钮

- 验证方式：模拟触发后观察气泡。

#### T1.4 桌宠设置入口

- 完成标准：设置可持久化并影响桌宠。
- 验证方式：重启后设置仍生效。

### M2：Coach 事件与本地规则触发

#### T2.1 定义 CoachEvent 数据模型

- 目标：为后续规则提供统一事件结构。
- 涉及文件（新增）：
  - `electron/coach/types.ts`
- 实现要点：
  - `CoachEvent`：
    - `event_id`
    - `session_id`
    - `event_type`：`idle_too_long / multiple_wrong / same_error / review_due / long_session / first_ac / boundary_suspected / complexity_warning`
    - `severity`
    - `evidence`
    - `created_at`
  - `ProblemSession`：
    - `session_id`
    - `problem_id`
    - `platform`
    - `started_at`
    - `last_active_at`
    - `active_seconds`
    - `submit_count`
    - `wrong_count`
    - `current_status`
    - `detected_stuck_level`
- 完成标准：类型可被各模块引用。
- 验证方式：tsc 通过。
- 优先级：P0。

#### T2.2 实现 ProblemSession 追踪

- 目标：在主进程维护“当前题目会话”。
- 涉及文件（新增）：
  - `electron/coach/ProblemSessionTracker.ts`
- 实现要点：
  - 监听 TabManager active tab 变化与 `problem:detected`。
  - 进入题目页时开启 session；切出到本地 IDE/其他应用时 session 挂起保持（不立即关闭），回站或提交仍归入同一会话。
  - `active_seconds` 只累计有效活跃时间：结合主窗口 focus/blur 与系统空闲检测（`powerMonitor.getSystemIdleTime()`），挂机不计时。
  - 区分三态：读题（题面页活跃）/ 写码（失焦但会话未结束）/ 卡壳（多次 WA 或长时间无进展后回访），`detected_stuck_level` 由三态推导——页面停留 ≠ 解题时间，误判卡壳比漏判伤害大。
  - 停留时间按 30-60s 聚合，不记录每次按键。
  - 暴露 `getCurrentSession()` / `getSessionHistory()`。
- 完成标准：能拿到当前题目会话与停留时长，且 IDE 失焦/挂机不虚增 `active_seconds`。
- 验证方式：日志打印 session 状态。
- 优先级：P0。

#### T2.3 订阅实时提交结果

- 目标：把 `submissions:detected` 作为 Coach 事件源。
- 涉及文件（新增）：
  - `electron/coach/CoachEventBridge.ts`
- 实现要点：
  - 复用现有 `RealtimeSubmissionService`：`SubmissionWatcher` 目前只 `webContents.send('submissions:detected')` 发给主窗口 renderer，主进程无订阅点，需为其增加主进程侧监听出口（回调注册或 EventEmitter），`CoachEventBridge` 订阅该出口。
  - 不修改现有抓取与解析逻辑，只加通知出口。
  - 每次提交结果转为 CoachEvent。
  - 同一题多次 WA/TLE/RE 触发 `multiple_wrong`。
  - 相同 verdict 重复出现触发 `same_error`。
- 完成标准：提交后 Coach 能感知。
- 验证方式：模拟提交触发事件。
- 优先级：P0。

#### T2.4 实现本地规则引擎

- 目标：根据事件 + 会话判断是否触发干预。
- 涉及文件（新增）：
  - `electron/coach/rules/RuleEngine.ts`
  - `electron/coach/rules/rules.ts`
- 实现要点：
  - 规则表：条件 + 评分 + 干预等级。
  - 示例规则：
    - 有效活跃时间超阈值且未提交 → `idle_too_long`（阈值随题目难度自适应：如 rating ≥ 1600 放宽到 20-30 分钟；卡题是正常训练，宁可漏报不误报）
    - 同题 WA >= 2 → `multiple_wrong`
    - 同题相同 verdict 重复 → `same_error`
    - 连续学习 > 90 分钟 → `long_session`
  - 节流：同类型事件 30 分钟内不重复触发。
  - 防 hint abuse：提示升级设冷却（每级间隔 ≥ 2 分钟或需一次新提交），不可零成本连点到底。
  - 比赛模式（T2.6）下规则引擎硬关闭。
  - 用户“今天别提醒”会临时屏蔽。
- 完成标准：规则可配置、可禁用。
- 验证方式：单元测试覆盖核心规则。
- 优先级：P0。

#### T2.5 IPC 注册与 preload 暴露

- 目标：按项目约定接入 Coach IPC。
- 涉及文件：
  - `electron/ipc/registerCoachIpc.ts`（新增）
  - `electron/ipc/registerMainIpc.ts`（接入）
  - `electron/preload.ts`（暴露）
- 实现要点：
  - channel 命名：`coach:getState` / `coach:triggerHint` / `coach:dismissHint` / `coach:feedback` / `coach:getSession`。
  - 遵循现有 getter 注入模式。
- 完成标准：renderer 可调用。
- 验证方式：tsc + 手动调用。
- 优先级：P0。

#### T2.6 比赛模式与合规硬保障

- 目标：rated 比赛进行中 Coach 绝对静默、合规可证明。CF 2024-09 规则禁止赛中使用 AI 产生解题思路，2025 年社区作弊风波后这是账号级红线——本任务是上线前置条件，也是差异化卖点（最直接竞品 Repovive 都做了赛时自动禁用）。
- 涉及文件（新增）：
  - `electron/coach/ContestGuard.ts`
- 实现要点：
  - 比赛检测：基于现有 parsers/sites 的 URL 规则判断当前 tab 是否处于进行中的比赛（CF `/contest/{id}`、洛谷比赛页），辅以比赛时间窗校验。
  - 检测到比赛进行中：规则引擎硬关闭、LLM 通道禁用、桌宠切换到明示的“比赛模式”状态（零提示、零气泡）。
  - 默认开启，不可通过普通设置绕过。
  - 审计日志：记录比赛时段与“零介入”事实（写入 `coach_interventions`），可导出作为诚信证明。
  - 赛后自动恢复，并可提示进入复盘/upsolve（仅赛后）。
- 完成标准：进入进行中的比赛页面后无任何提示触发，审计日志可导出证明。
- 验证方式：单元测试覆盖 URL/时间窗判定 + 手动进入比赛页观察。
- 优先级：P0（上线前置条件）。

### M3：通用提示库与低 token 交互

#### T3.1 建立通用提示模板库

- 目标：无大模型时也能给出有价值的小提示。
- 涉及文件（新增）：
  - `electron/coach/hints/hintTemplates.ts`
- 实现要点：
  - 分类：
    - 复杂度类：“这道题的时间复杂度能通过吗？数据范围 1e5 通常需要 O(n log n) 以内。”
    - 边界类：“边界处理好了吗？空数组、单元素、n=1 都试过吗？”
    - 数据范围类：“注意变量类型，1e9 以上要用 long long / BigInt。”
    - 初始化类：“数组/累加器初始化值是否正确？是否需要重置？”
    - 溢出类：“中间结果会溢出吗？乘法累加要小心。”
    - 输入输出类：“输入输出格式对了吗？多组数据、行尾空格？”
    - 特判类：“特殊情况是否特判？n=0、负数、相同元素。”
    - 数组越界类：“下标从 0 还是 1 开始？访问前是否检查边界？”
    - 循环条件类：“循环边界是 < 还是 <=？是否漏掉最后一个元素？”
  - 每条提示带 `id` / `category` / `tags?` / `text`。
  - MVP 阶段为 TS 内置模板文件（不入库）；M7 时再迁移为 SQLite 表 `hint_templates`。
- 完成标准：至少 30 条覆盖常见场景的模板。
- 验证方式：人工评审。
- 优先级：P0。

#### T3.2 提示选择策略

- 目标：根据事件类型选合适提示。
- 涉及文件（新增）：
  - `electron/coach/hints/HintSelector.ts`
- 实现要点：
  - `multiple_wrong` → 优先边界/越界/循环类。
  - `TLE` → 复杂度类。
  - `RE` → 越界/初始化类。
  - `WA` → 边界/特判/输入输出类。
  - `idle_too_long` → 元认知类：“先想清楚再写，能不能举几个例子？”
  - 不依赖算法标签。
- 完成标准：不同 verdict 产生差异化提示。
- 验证方式：单元测试。
- 优先级：P0。

#### T3.3 分级提示策略

- 目标：实现 Socratic Ladder。
- 涉及文件（新增）：
  - `electron/coach/hints/HintLadder.ts`
- 实现要点：
  - 阶梯按算法竞赛特点排序：概念/标签在 CP 中“想法即答案”，比实现细节更剧透，故放最高层（原设计照搬 CS1 场景，已修正）。
  - Level 0：不提示，仅记录。
  - Level 1：轻提醒（“要不要看一个方向提示？”）。
  - Level 2：元认知（“先想清楚状态能否被复用？举几个小例子试试？”）。
  - Level 3：关键细节/边界（“注意下标从 0/1 开始时边界不同”——不涉及算法思想）。
  - Level 4：策略（“维护区间累计信息，O(1) 查询”）。
  - Level 5：概念/标签（“这题可能和前缀和有关”——仅当有标签，且升级前需二次确认“该提示接近题解方向”）。
  - 用户每次点“再给一点”才升级，不直接跳级；升级有冷却（见 T2.4 防 hint abuse）。
- 完成标准：可控制升级流程。
- 验证方式：手动演练。
- 优先级：P1。

#### T3.4 用户反馈与节流

- 目标：记录用户反馈用于后续优化。
- 涉及文件（新增）：
  - `electron/coach/CoachFeedbackStore.ts`
- 实现要点：
  - 反馈类型：`helpful / not_helpful / dismiss / never_today`。
  - 写入 SQLite（建议表 `coach_feedback`，迁移在 M2 后补）。
  - 反馈影响后续同类型提示频率。
- 完成标准：反馈可持久化。
- 验证方式：重启后历史反馈仍存在。
- 优先级：P1。

#### T3.5 本地题面约束解析（零 LLM 靶向提示）

- 目标：从题面自动抽取数据范围与时限，把通用提示变成靶向提示——“n ≤ 2e5 通常需要 O(n log n) 以内，检查嵌套循环”这类提示是评委眼中“真技术”的核心演示点，也是浏览器扩展难以全站做到的能力。
- 涉及文件（新增）：
  - `electron/coach/problemFacts/ConstraintParser.ts`
- 实现要点：
  - 通过内嵌浏览器注入脚本抽取题面约束文本，正则解析 `1 ≤ n ≤ 2·10^5`、`时间限制 1.00s` 等模式（CF/洛谷先行，复用现有 parsers 站点扩展点）。
  - 与 verdict 联动：TLE → 数据范围与复杂度对照提示；WA 且值域 ≥ 1e9 → 溢出提示。
  - 解析失败静默退化到通用提示，不阻塞主流程。
  - 解析结果可缓存（`026_problem_constraints.ts`，可选）。
- 完成标准：CF/洛谷题面能抽出 n 范围与时限，TLE 提示中出现具体数值对照。
- 验证方式：对 20 道样例题面统计抽取准确率（目标 ≥ 80%）。
- 优先级：P0。

### M4：用户主动上传代码分析入口

#### T4.1 代码上传 UI

- 目标：用户卡住时主动粘贴/上传代码。
- 涉及文件（新增）：
  - `src/features/coach/CodeAnalysisPanel.tsx`
- 实现要点：
  - 入口：桌宠气泡“分析我的代码”按钮。
  - 文本框 + 文件上传。
  - 显著隐私提示：“代码不会被自动上传，仅在你确认后发送到 LLM。”

## 5. 推荐实施顺序

1. **M1 视觉原型**：先把桌宠做出来，验证科技感与体验。
2. **M2 事件触发 + 比赛模式**：接入实时提交结果，T2.6 合规硬保障同批落地。
3. **M3 通用提示库 + 题面约束解析**：无 LLM 也能给出靶向提示。
4. **M9 过程复盘与导出**：打动真实用户（选手/教练）的核心功能，提前做。
5. **M5 形象自定义**：差异化重点。
6. **M6 LLM Provider**：可选增强。
7. **M4 代码分析**：依赖 M6。
8. **M7 知识库**：内容积累后做。
9. **M8 复赛创新**：最后打磨。

## 6. 不做事项

- 不默认上传用户代码。
- 不直接给完整题解。
- 不依赖算法标签作为唯一判断。
- 不把 Coach 做成普通聊天框。
- 不在 MVP 阶段强制接 Live2D。
- 不在后台静默采集源码。
- 不在 renderer 直接访问 SQLite。
- 不复用 `aiOutputRepository` 之外的核心事实表写入权限。
- 不在 rated 比赛进行中提供任何提示、分析或 LLM 调用（T2.6 硬保障）。

### 强烈建议采纳（低成本 × 高独特性）

#### C1 隐形弃题挖掘与回收（评分 9，工作量低）

- 一句话：捕捉“读了几分钟就悄悄放弃、从未提交”的题，自动生成竞品看不见的隐形错题本，按投入时间 × 薄弱标签排序，经复习计划择时重推。
- 依托：现有 `problem_visits` 停留 + `submissions` 反连接查询 + `reviewPlanner`，近乎纯 SQL。
- 新颖性核查：未找到同类产品（最接近的 CF Tracker 等扩展只计时、不判定弃题、不回收复习）。宣传口径：说“当前没有任何工具做到”，不说“只有内嵌浏览器能做到”（浏览器扩展理论上也可拿到该信号）。
- 建议位置：M3 尾部新增任务。

#### C2 AC 时刻自我解释卡（评分 8，工作量低）

- 一句话：AC 瞬间弹卡追问“从 WA 到 AC 你改对了什么”，卡片自动带“WA on test 3 → AC”上下文，答案挂题入库，复习时先考这句再放题面（自我解释效应，学习科学 30 年背书）。
- 依托：实时 verdict 流（AC 且此前有 WA 或看过题解才触发）+ 现有笔记/复习计划表。
- 新颖性核查：未找到“AC 瞬间自动触发自我解释”先例；SIGCSE'26 的 Owlgorithm 做 CP 自我解释但需手动上传代码自报结果。宣传口径：称“首个自动时机触发”，不称“无人做过 CP 自我解释”。
- 建议位置：M3 尾部或 M9。

### 建议采纳（答辩核心武器）

#### C3 信号驾驶舱与白盒证据链（评分 8，工作量中）

- 一句话：DevTools 式面板实时投屏“信号流瀑布 → 规则状态机 → 每条提示的证据链”，任一提示可回溯到 SQLite 原始数据行——当场自证不是套壳 GPT。
- 依托：主进程事件总线已有全部信号；干预落库时附带触发信号 ID 链。
- 新颖性核查：可解释 ITS（XAI）与 LLM 观测工具（Langfuse 类）分别存在，本场景组合无先例。答辩勿称要素首创，称“组合 + 场景”。
- 建议位置：M9（与 T9.4 相邻，答辩强依赖）。

#### C4 题解消化质量闭环（评分 7，工作量中）

- 一句话：区分题解是“扫一眼 / 精读 / 复制粘贴”（copy 监听 + 题解页停留分层 + 读后首提延迟），复制型 AC 自动排入 3 天后盲写复验队列。
- 依托：WebContentsView preload 注入 copy 监听 + T9.3 的题解页识别 + `reviewPlanner`。
- 新颖性核查：粘贴检测、题解访问标记、行为驱动复习各自成熟，串成闭环无竞品。直击教练“查抄题解”最大痛点（评审中教练最想要的能力）。
- 建议位置：M8/M9。

### 可选（演示戏剧性强，宣传口径需收窄）

#### C5 宠物押题：可校准的 AC 直觉（评分 8，工作量中）

- 一句话：提交瞬间宠物押出 AC 概率并演出候判忐忑，verdict 翻牌兑现，长期展示校准曲线；检测“赌气狂交”时温柔拦截。本地逻辑回归在线训练，完全离线。
- 依托：提交间隔/停留/同题 WA 串/标签历史胜率做特征，verdict 流天然当训练标签。
- 新颖性核查：提交结果预测（学术）、乱交检测（Scooter the Tutor 2006）、编辑器宠物各有先例；“提交时校准押题”的组合无先例。按“组合创新”宣传；校准曲线可正面回应“if-else 包装成 AI”的质疑。

#### C6 逐次提交代码快照与差分分析（评分 8，工作量中）
